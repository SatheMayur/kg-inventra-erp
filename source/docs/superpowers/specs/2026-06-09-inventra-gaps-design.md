# Inventra Gap Build — Design Spec
**Date:** 2026-06-09
**Author:** Mayur Sathe (Developer & Project Lead)
**Scope:** Add 4 missing subsystems to Inventra (`source/`, Next.js App Router + Prisma) WITHOUT duplicating existing features. Reuse existing models, `mutateStock`/`releaseReservation` helpers, `ApiError`/`handleApiError`, Zod, `authorize`, `createAuditLog`.

## Guiding constraints
- No duplicated feature or function. Extend existing models/routes; add new only where no equivalent exists.
- Every stock change still flows through the single `mutateStock` ledger helper.
- All writes: Zod-validated, role-checked, audit-logged, transactional.
- One Prisma migration covers all schema changes.

---

## Subsystem A — Request workflow

**Goal:** department-head approval routing + a "ready for pickup" stage.

**Schema**
- `User.isDeptHead Boolean @default(false)`.

**Behavior**
- Approver resolution: a request is approvable by (a) any `admin`, or (b) a user with `isDeptHead = true` whose `department` equals the requester's `department`. Otherwise 403.
- Request status lifecycle extended: `Pending → Approved → ReadyForPickup → Issued`, plus `Rejected`, `Cancelled`.
  - `ReadyForPickup` set by store staff (admin) after `Approved`, before `Issued`.
  - Stock reservation already happens at request creation; issue still deducts via `mutateStock`. No change to stock math.

**Interfaces**
- `PATCH /api/requests/[id]/approve` — add dept-head authorization check (reuse existing handler; replace admin-only gate with the resolver).
- `PATCH /api/requests/[id]/ready` — new: Approved → ReadyForPickup (admin/dept-head).
- Issue route: only allow from `Approved` or `ReadyForPickup`.
- UI: requests/issuance views show the new status; a "Mark ready" action.

## Subsystem B — Procurement automation

**Goal:** auto-create POs on low stock + budget-limit approval.

**Schema**
- `Item.preferredSupplierId String?` (FK Supplier), `Item.reorderQty Int @default(0)`.
- `PurchaseOrder.status` gains `PENDING_APPROVAL`; add `approvedBy String?`, `approvedAt DateTime?`, `autoGenerated Boolean @default(false)`.
- Reuse `FeatureFlag`/settings for `PO_APPROVAL_LIMIT` (numeric, stored as a settings row).

**Behavior**
- New helper `lib/reorder.ts → checkReorder(tx, itemId)`: after a stock-decreasing op, if `(stock - reservedQty) <= minStock`, `reorderQty > 0`, `preferredSupplierId` set, and no open (`DRAFT|PENDING_APPROVAL|SENT`) PO already covers the item → create a `DRAFT` PO line for `reorderQty`. Called from the stock-out paths that use `mutateStock` (issue, checkout, transfer-confirm) — single call site each, no duplicated logic.
- Budget approval: when a PO is moved toward `SENT`, if `totalAmount > PO_APPROVAL_LIMIT` it must pass through `PENDING_APPROVAL` and be approved by an admin (sets `approvedBy/approvedAt`). Under the limit → may go straight to `SENT`.

**Interfaces**
- `POST /api/purchase-orders/[id]/approve` — new (admin): PENDING_APPROVAL → approved/SENT.
- Existing PO send/receive flow extended with the status guard. Receive remains the already-fixed atomic claim.

## Subsystem C — IT Asset management (new module)

**Goal:** serialized assets assigned to individuals; warranty/license tracking.

**Schema (new model)**
```
Asset {
  id, name, serialNumber @unique, itemId? (FK Item, optional catalog link),
  status: IN_STOCK | ASSIGNED | MAINTENANCE | RETIRED (default IN_STOCK),
  assignedToUserId? (FK User), assignedAt?,
  warrantyExpiry?, licenseExpiry?, notes?, createdAt, updatedAt
}
```
- Extend `MaintenanceSchedule` with `assetId String?` (reuse the existing maintenance module for assets too — no new maintenance engine).

**Behavior**
- Distinct from bulk-stock `Item` (quantity-based) and from short-term `ItemCheckout` (loaner). An Asset is one physical unit with a serial, owned long-term by an employee.
- Assign: IN_STOCK → ASSIGNED (set user + assignedAt), audit-logged. Return: ASSIGNED → IN_STOCK.

**Interfaces**
- `GET/POST /api/assets`, `GET/PATCH/DELETE /api/assets/[id]`, `POST /api/assets/[id]/assign`, `POST /api/assets/[id]/return`.
- UI: Assets view (list, assign dialog, status badges).

## Subsystem D — Warranty/license alerts + Supplier performance

**Goal:** surface upcoming expiries; measure supplier delivery + accuracy.

**Schema**
- `PurchaseOrder.expectedDeliveryDate DateTime?`, `PurchaseOrder.receivedAt DateTime?` (set on receive).
- `POItem.receivedQty Int @default(0)` (set on receive; enables accuracy).

**Behavior**
- Warranty/license alerts: extend the existing alerts route to include Assets with `warrantyExpiry`/`licenseExpiry` within N days. Reuse the alerts pattern — no new engine.
- Supplier performance report: per supplier — avg delivery delay (`receivedAt − expectedDeliveryDate`), fulfillment accuracy (`sum(receivedQty)/sum(qty)`), PO count. Read-only aggregation.

**Interfaces**
- `GET /api/reporting/supplier-performance` — new.
- Receive flow records `receivedAt` + per-line `receivedQty` (extends the already-fixed receive handler).
- UI: supplier-performance report view; warranty alerts surfaced in existing alerts UI.

---

## Cross-cutting
- **Migration:** one `prisma migrate` adding all fields/models above.
- **Auth:** new write routes admin-only except request approve/ready (admin or matching dept head).
- **Audit:** assign/return, PO approve, auto-PO creation, ready-for-pickup all call `createAuditLog`.
- **Testing:** unit tests for `checkReorder` (threshold logic), approver resolver, supplier-performance math; route tests for asset assign/return and PO approval state machine.

## Out of scope (explicitly not built)
- QuickBooks / external accounting integration.
- Multi-level (>1) approval chains.
- Asset depreciation accounting.

## Build order (if executed incrementally)
A (request) → B (procurement) → C (assets) → D (reports/alerts).
